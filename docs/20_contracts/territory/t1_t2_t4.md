# T1/T2/T4 国度步骤契约

## 阶段边界

| 阶段 | 目的 | 前置 | 主要产物 |
| --- | --- | --- | --- |
| T1 | 基于 W4 世界数据为每个国度生成大陆候选簇，并等待确认最终簇 | W4 已完成，`regionCacheMap` 有目标大陆 detail cache | `T1_Candidates.json`、`T1_Status.json`、大陆级 `T1_Index.json` |
| T2 | 基于 T1 已选簇生成方向候选，确认首都方向/点位并写入 territory config | T1 已选簇 | `T2_DirectionCandidates.json`、`T2_Status.json`、T2 首都摘要 |
| T4 | 基于当前实例已存在的 T3 主权结果扫描国度内部地形事实，供 C1/C 阶段读取 | 当前实例已具备 claimed/wild chunks，可来自 T3 运行或 `/territory/t3_import` 导入 | 每个国度的 T4 summary/dat、T4 batch report |

T3 的单大陆扩张入口已在 `docs/20_contracts/territory/t3_run_continent.md` 单独定义。

## T1 HTTP 入口

### `GET/POST /t1_blueprint`

- `POST` 保存原始 blueprint。
- `GET` 列出当前 blueprint。
- 响应 `step` 为 `T1`。

### `POST /t1_candidates_for_continent`

请求：

```json
{
  "continent_id": 1
}
```

响应核心字段：

```json
{
  "continent_id": 1,
  "territories": [
    {
      "territory_id": "han",
      "territory_instance_id": "han@c1",
      "territory_name": "Han",
      "continent_id": 1,
      "workflow_state": "pending_ai_selection",
      "blocked": false,
      "cluster_selected": false,
      "candidates_bundle": {}
    }
  ]
}
```

### `POST /t1_select_cluster`

请求必须包含 `territory_id`、`continent_id`，并提供 `cluster_id` 或 `label` 之一。

```json
{
  "territory_id": "han",
  "continent_id": 1,
  "cluster_id": 2
}
```

成功后 `T1_Status.json` 中 `clusterSelected=true`，`workflowState=done`。

## T2 HTTP 入口

### `POST /t2_direction_candidates`

请求：

```json
{
  "territory_id": "han",
  "continent_id": 1
}
```

响应包含 `workflow_state / blocked / recommended_adjustments`，以及可选 `direction_bundle`。

### `POST /t2_select_direction`

请求：

```json
{
  "territory_id": "han",
  "continent_id": 1,
  "direction": "north"
}
```

成功后必须：

- 写入 `T2_Status.json`，且 `pointSelected=true`。
- 通过 `TerritoryManager.createTerritory(...)` 写入运行时 territory config。
- 导出 T2 capital summary，供 `/territory/summary` 读取。

## T4 HTTP 入口

### `POST /workflow/run`

通过工作流运行 T4：

```json
{
  "stage_id": "T4",
  "sample_stride": 2,
  "max_chunks_per_territory": -1,
  "loaded_only": false
}
```

请求兼容规则：

- `stage_id` 与 `stageId` 都可接受；缺少 stage 时返回 `400`，不会再静默默认跑 T1。
- T4 运行参数同时接受文档命名与内部命名：
  - `sample_stride` 或 `t4_sample_stride`
  - `max_chunks_per_territory` 或 `t4_max_chunks`
  - `loaded_only` 或 `t4_loaded_only`

T4 运行选项：

- `sample_stride`：1 到 16，默认 2。
- `max_chunks_per_territory`：大于 0 时限制每个国度扫描 chunk 数；`-1` 表示不限制。
- `loaded_only`：为 true 时只扫描已加载 chunk，适合自动触发轻量模式。

显式调用 `/workflow/run` 执行 T4 时，不会隐式重跑 T3；如果当前实例还没有有效面积，应先跑 T3 或调用 `/territory/t3_import` 恢复当前实例主权。

### `GET /territory_status`

返回当前注册国度的运行态主权摘要。入口会尝试把当前实例 id 的 T3 结果从磁盘恢复到内存，因此可用来确认重启后的自动恢复是否成功。

### `GET /territory/summary`

查询参数：

- `territoryId`：必填，可为基础国度 id。
- `continentId`：可选，用于解析 `territoryId@c{continentId}` 实例 id。

### `POST /territory/t4_window`

请求：

```json
{
  "territory_id": "han@c1",
  "center_x": 3200,
  "center_z": -3200,
  "radius_blocks": 256,
  "max_points": 160,
  "max_biome_samples": 3000
}
```

成功响应 `step=T4`，并返回窗口内 `terrain / biome_composition_window / samples`。缺少 T4 dat 时返回 `404`。

窗口中心规则：

- 显式传入 `center_x/center_z` 时，按请求中心取窗口。
- 未传 `center_x/center_z` 时，默认先用 territory capital。
- 如果默认 capital 落在 T4 覆盖区之外，接口会自动吸附到最近的 T4 采样点，并在响应中返回：
  - `warnings=["requested_center_outside_t4_coverage","center_reanchored_to_nearest_t4_cell"]`
  - `window.reanchored=true`
  - `window.requested_center_x / requested_center_z`
  - `window.nearest_distance_blocks`

## 下游约束

- T1/T2 不运行 T3。
- T2 不应在 T1 未确认簇时创建国度。
- T4 不改变国度边界，只读取 T3 主权结果并补地形事实。
- T4 可以读取通过 `/territory/t3_import` 导入到当前实例 id 的历史 T3 结果。
- C1 前应先检查 `/territory_status` 中目标国度 `total_area > 0`，再读取 `/territory/t4_window`。
