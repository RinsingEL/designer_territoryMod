# AI 协作开发规范

## 定位

本文档是本项目通用开发协作规范，不只适用于 C 大重构。

它补充 `AGENTS.md` 的仓库和文档规则，用来约束每次进入实现开发时，AI 和人工如何对齐目标函数、代码设计、验收边界和人工介入时机。

核心目标：

- 避免 AI 在错误目标函数上持续调试。
- 避免修一个 bug 时把已确认的代码逻辑回退或开倒车。
- 让非视觉、非设计类问题尽量由 AI 自测自修闭环。
- 让设计方向、代码目标、审美效果和阶段边界及时交给人工评估。

## 开发前设计检查单

每个实现任务开始前，AI 必须先给出短设计检查单。用户确认后再进入代码实现。

```text
本阶段真正要解的问题：
输入：
输出：
不做的事情：
成功标准：
失败时怎么回退：
准备采用的算法或代码路径：
需要用户评估的点：
```

这份检查单必须优先回答“现在到底在解什么问题”。例如：

- 是在解 jigsaw 方块连接关系，还是在找结构 placement。
- 是在识别 image2 已画出的道路草图，还是在重新规划道路。
- 是在转换数据契约，还是在改变阶段职责。

如果这些问题没有对齐，不应进入代码实现。

## 代码设计闸门

以下情况必须先停下来给用户评估，不能直接实现：

| 情况 | 为什么需要人工评估 |
| --- | --- |
| 阶段职责变化 | 可能改变 C1-C9 或 T/W 阶段的系统边界 |
| 核心算法目标变化 | 容易出现名词相同但目标函数不同的问题 |
| 数据真值变化 | 会影响后续所有调试和验收依据 |
| fallback 策略变化 | 可能绕过原目标，变成曲线救国 |
| 旧逻辑被替换或回退 | 容易修 A bug 时让 B 设计开倒车 |
| 产物表现由“能运行”变成“是否合理/好看” | 需要用户审美和方案判断 |
| AI 准备新增大量补丁逻辑 | 可能说明方向错了，而不是 bug 复杂 |

如果实现中发现原设计不成立，AI 应先说明：

```text
原设计假设：
实际发现：
继续按原方案的风险：
建议改法：
需要用户判断的问题：
```

## AI 可自行闭环的问题

以下问题默认由 AI 自行修复、补日志、补测试、重新跑验证，不需要每次打断用户：

| 类型 | 示例 |
| --- | --- |
| 代码错误 | 编译失败、测试失败、空指针、路径错误 |
| 契约错误 | 字段名不一致、必填字段缺失、版本号错误 |
| 坐标与几何 bug | 坐标转换偏移、AABB 不对、overlay 与 JSON 不对齐 |
| 稳定性问题 | 固定输入多次运行结果不一致 |
| debug 产物明显不合理 | 道路跑到城市外、功能区全碎、graph 不连通 |
| 小范围工程修复 | 端点吸附、噪点清理、节点去重、报告字段补齐 |

AI 自行闭环时，必须保留可复查证据：

- 结构化产物。
- debug / preview 产物。
- 测试命令与结果。
- 修改记录或验收摘要。

## 必须人工介入的问题

以下问题不能由 AI 独自决定：

| 类型 | 示例 |
| --- | --- |
| 目标函数不确定 | jigsaw 解算 vs structure placement |
| 视觉或设计效果 | 城市外形、道路大形、功能区布局是否满意 |
| 阶段边界争议 | C4 能否补路、C3 能否改功能区语义 |
| 多轮修复不收敛 | 修了几轮又回到老问题 |
| 曲线救国倾向 | 为了让测试过，绕开 image2、绕开 jigsaw 或绕开阶段真值 |
| 方案体验判断 | 产物能用但不像用户想要的效果 |
| 大范围重构 | 新增核心抽象、替换主链、改变数据流 |

人工介入时，AI 必须提供最小可判断材料，而不是只描述感觉：

- 当前产物路径。
- 对比截图或 overlay。
- 当前 JSON / 报告摘要。
- 自动测试结果。
- AI 自己判断哪里偏了。
- 需要用户只判断的 1-3 个问题。

## 自修轮次上限

同一类问题最多允许 AI 自修两轮。

第三次仍不收敛时必须暂停，并输出：

```text
问题现象：
已尝试修复：
为什么没有收敛：
当前 debug 产物：
我怀疑的设计偏差：
需要用户判断：
```

这条规则用于防止 AI 在错误方向上长时间调试，尤其适用于：

- C7-C9 结构放置与 jigsaw。
- image2 / CV 解析。
- 道路、parcel、功能区视觉效果。
- fallback 和兼容逻辑。

## 每轮交付三件套

每个阶段进入验收时，AI 必须交付三件套。

| 交付物 | 说明 |
| --- | --- |
| 结构化产物 | JSON、dat、契约对象、状态报告 |
| debug / preview 产物 | overlay、截图、日志、trace、识别报告 |
| 验收摘要 | 自动验证了什么、AI 人工看了什么、还需要用户判断什么 |

如果缺少 preview 或 debug 产物，AI 不能只凭“测试通过”要求用户验收视觉/设计类任务。

## 不允许的行为

- 不允许在用户未确认时，把阶段目标从 A 偷换成 B。
- 不允许为了让测试通过，绕过文档真值或阶段真值。
- 不允许把 fallback 写成默认主链。
- 不允许修 bug 时删除、回退或覆盖用户已确认的设计逻辑。
- 不允许把“能 place 一个结构”当成“jigsaw 解算正确”。
- 不允许把“CV 解析成功”当成“城市设计满意”。
- 不允许把“多模态觉得可以”当成“像素级结构化真值”。

## 与文档仓库的关系

当本规范触发以下情况时，AI 必须同步判断是否更新文档仓库：

- 阶段职责变化。
- 数据契约变化。
- 验收方式变化。
- 测试入口变化。
- 影响面变化。
- 代码导览入口变化。

正式方案和契约必须写入 `docs/10_product`、`docs/20_contracts`、`docs/30_code_guide` 或 `docs/40_tests`，不能只留在实现仓库 `dev_docs`。
