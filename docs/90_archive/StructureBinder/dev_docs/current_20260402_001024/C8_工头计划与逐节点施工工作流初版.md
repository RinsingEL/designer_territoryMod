# C8 工头计划与逐节点施工工作流初版

## 1. 目标

当前 `C8` 不再沿“排列方式枚举 / 抽象求解器”路线继续加层，而是改成一套更贴近施工流程的最小工作流：

- 总规阶段只产出一份工头计划
- 当前只允许一个 `start`
- 后续所有结构都通过节点施工队列逐步扩出
- 每次只处理一个节点
- 程序负责落地校验、状态维护、错误回包和子节点入队
- AI 负责在当前节点上下文内选方向、选模板、根据失败原因重试

这套方案的核心目标不是一次给出完整排布结果，而是让每个节点都能在干净局部上下文里做可解释的施工决策。

## 2. 总规阶段：工头计划

### 2.1 职责

工头计划只做指导，不直接给完整施工结果。

它负责：

- 给出全局建筑方向和总目标
- 指定唯一的 `start`
- 给出若干 `phase`
- 为每个 `phase` 写清当前阶段优先修什么、暂时不要碰什么
- 给后续节点施工提供总原则与必要的全局限制

它不负责：

- 不直接给完整排布图
- 不提前写死所有未来节点
- 不直接确定所有节点最终模板
- 不承担多 `start` 并行调度

### 2.2 当前建议结构

工头计划当前先保持很轻，只需要包含：

- `global_goal`
- `start_node`
- `phase_list`
- `global_constraints`
- `foreman_notes`

其中：

- `start_node` 只保留一个
- `phase_list` 更像施工顺序指导，而不是复杂状态机
- `foreman_notes` 可以用自然语言描述整体框架和关键禁忌

### 2.3 phase 的定位

`phase` 需要保留，但职责是“施工引导顺序”，不是建筑学分类。

它适合表达：

- 先修中轴线和前庭
- 再修两侧附属部分
- 最后修外围花园与收尾节点

它不适合承载：

- 复杂求解器参数
- 一大套抽象语义枚举
- 多工区并行依赖图

## 3. 节点施工阶段

### 3.1 基本原则

后续不再给图，只给结构化数据。每次只让一个 subagent 处理一个节点。

当前节点的 subagent 不负责定义“我要修什么”，而是接收一个已经明确的施工目标，然后在合法范围内把它修出来。

### 3.2 节点进入施工时，AI 已知信息

当一个节点进入施工时，AI 至少应当知道：

- 当前处于哪个 `phase`
- 工头对当前阶段的指导计划
- 此前已经建成的所有建筑列表
- 父节点信息
- 当前节点是什么结构
- 当前约束，允许为空
- 已失败记录

另外：

- 地形信息不必全部提前塞进上下文，可由接口按需获取
- 当前节点可走哪些拼图方块，需要由程序明确提供
- 每个拼图方块下可选哪些模板，直接使用该拼图方块自带模板池

### 3.3 节点结构来源

节点是什么结构，不由当前 subagent 临时猜。

它由两部分共同约束：

- 当前 `phase` 计划提供施工方向和倾向
- 当前拼图方块自带模板池提供实际可选范围

AI 只在这两者交集里做选择，不允许跳出模板池发明新结构。

### 3.4 单次节点施工动作

当前节点的 subagent 需要自己决定：

- 走哪个拼图方块方向
- 在该拼图方块模板池里选哪个结构模板
- 是否根据失败原因继续重试

父节点并不替它把方向提前定死，只规定链路来源和施工目标。

## 4. 程序与 AI 的职责边界

### 4.1 AI 负责

- 结合 `phase` 指导选择当前节点优先尝试哪个拼图方块
- 在当前拼图方块模板池内选择结构模板
- 根据失败回包判断是否换模板、换方向或终止
- 在必要时放宽当前结构的地形落地限制后再试

### 4.2 程序负责

- 维护工头计划、节点状态和施工队列
- 提供当前节点可走的拼图方块及其模板池
- 调用地形接口并返回地形相关信息
- 尝试实际落地
- 返回结构化失败原因
- 统计每个拼图方块已尝试次数
- 在节点成功后生成新的子节点并入队
- 在所有尝试用尽后结束该节点

## 5. 失败重试规则

### 5.1 错误回包要求

程序返回的失败信息必须详细，不能只给一句“放置失败”。

至少要能让 AI 判断：

- 为什么失败
- 失败发生在哪一步
- 是地形、对接、边界还是碰撞问题
- 当前拼图方块还剩多少可尝试空间

### 5.2 AI 失败后的允许动作

失败后，AI 可以：

- 在当前拼图方块下换另一个模板
- 放宽当前模板的地形落地限制后重试
- 放弃当前拼图方块，改试别的拼图方块方向

### 5.3 AI 失败后的禁止动作

失败后，AI 不允许：

- 改当前节点是什么结构
- 跳出当前拼图方块模板池乱选结构
- 改拼图连接契约
- 越界放置
- 忽略与已落地结构的硬碰撞
- 改写不是地形条件的硬限制

### 5.4 地形放宽规则

AI 唯一允许主动放宽的，是结构自身的地形落地限制。

例如：

- 地面平整度要求
- 局部坡度容差
- 局部高度差容差

除此之外的规则，本版都不开放给 AI 动态放宽。

### 5.5 拼图方块尝试上限

每个拼图方块都有自己的模板池。

如果某个拼图方块的尝试次数已经超过该模板池可尝试模板个数，则直接停止在这个拼图方块上的继续尝试，改试下一个拼图方块方向。

## 6. 节点终止与队列推进

### 6.1 节点内尝试顺序

当前节点的尝试顺序如下：

1. 查看当前节点所有可走拼图方块
2. 选择一个方向
3. 在该拼图方块模板池内逐个尝试
4. 失败时根据详细回包重试或换模板
5. 当前拼图方块试尽后，回到节点层换下一个拼图方块
6. 所有拼图方块都不可行时，结束该节点

### 6.2 节点成功后的队列规则

节点只要有成功落地结果：

- 程序就根据这次成功结果实际打开的新拼图方块
- 生成新的子节点任务
- 将子节点压入待施工队列

后续继续按队列逐个处理，不做并发施工。

### 6.3 节点失败后的终止规则

如果出现以下任一情况，则结束当前节点：

- 所有可走拼图方块都试过，仍无可落地方案
- 当前所有拼图方块本身都已是自然终止方向
- 当前节点没有任何成功落地结果，因此不再产生子节点

## 7. 当前版本的最小闭环

本版工作流可以先收敛成如下闭环：

1. 工头计划选择唯一 `start`
2. `start` 节点入队
3. subagent 读取当前节点上下文
4. subagent 选择拼图方块方向
5. subagent 在该方向模板池中选择结构
6. 程序尝试落地并返回成功或详细失败原因
7. 失败则继续由 AI 判断换模板、放宽地形条件或换方向
8. 成功则由程序生成子节点并入队
9. 节点全部方向穷尽则终止
10. 队列清空则本轮施工结束

## 8. 后续待补项

这份初版草案先解决最小流程，不在本轮展开以下内容：

- 正式 `json schema`
- planner prompt 与 repair prompt 全文
- validator error code 枚举表
- 多 `start` 和并发施工
- 复杂阶段迁移规则
- 子节点结构命名规范
- 与 `C7/C8/C9` 现有类和接口的一一映射

这些内容建议在本草案确认后，再按“接口层 -> prompt 层 -> 代码迁移 TODO”继续细化。
