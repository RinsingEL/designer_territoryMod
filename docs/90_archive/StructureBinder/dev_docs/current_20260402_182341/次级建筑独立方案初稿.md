# 次级建筑独立方案初稿

## 1. 背景

当前主模块建筑链路已经基本切到：

- `C7` 工头计划
- `C8` 节点决策与校验
- `C9` 队列生成与建造执行

在这条主链稳定后，下一阶段更适合优先推进次级建筑，而不是道路系统。

原因：

- 道路当前更倾向外包，不适合现在作为主开发方向深挖内部实现。
- 次级建筑可以直接承接当前“主体建筑先定调、剩余空间再填充”的既有口径。
- 仓库内已经存在关于“次级建筑后置补充”“次级建筑默认输出为可建造块”“次级建筑 `single` 化”的现成结论，适合继续落地。

## 2. 已有结论回收

本初稿沿用以下已讨论过的稳定口径：

- 次级建筑是后置补充层，不与主体建筑争夺核心语义。
- 主体建筑允许保留复杂 arrangement；次级建筑整体降级，不走同等复杂度流程。
- 当主体建筑最终有效落地面积占比 `<= 60%` 时，允许进入次级建筑补充流程；`> 60%` 时默认不强制补次级建筑。
- `C8` 对次级建筑的默认输出，不是复杂拼图扩张结果，而是一批“次级可建造块”。
- `C9` 再基于这些次级可建造块，按大小和约束随机或规则化落地次级结构。

## 3. 本案目标

本案不追求一次做成完整的次级建筑系统，而是先做一个最小闭环：

- 从主体建筑结果中识别需要补充的空间
- 生成一批次级可建造块
- 让大多数次级建筑按 `single` 路径落地
- 复用现有 `C9` 队列与执行链路
- 保证次级建筑不明显破坏主体建筑表达

## 4. 本案不做的事情

- 不把次级建筑重新拉回主体建筑那套“工头计划 + 逐节点复杂 AI 决策”主链。
- 不优先建设道路系统本体。
- 不在第一阶段强行支持大量 `pair / linear / start-expand` 例外。
- 不先接入店铺命名、NPC 背景、地图 POI 等高阶语义。

## 5. 建议主线

### 5.1 阶段定位

建议把次级建筑明确定位为：

- 主体建筑基本稳定后进入
- 作为后置空间填充层
- 默认不参与主体主骨架的关键决策

换句话说，次级建筑不是“主体施工过程中的并行节点群”，而更像“主体完成后的第二轮补全”。

### 5.2 输入来源

建议第一阶段优先使用现有 `C6` 的 `secondary_fill` 作为入口语义来源，再结合主体建筑最终结果生成次级可建造块。

现有代码中已经存在：

- `LayoutPlan.secondary_fill`
- `SecondaryFill { zone, style, params }`

因此不建议重新发明一个完全平行的新入口。

### 5.3 中间产物

建议在 `C8` 或 `C8 -> C9` 之间引入明确的“次级可建造块”概念，作为次级建筑的核心中间产物。

每个次级可建造块建议至少包含：

- `build_area_id`
- `group_id`
- 块的范围或 rect
- 尺寸信息
- 所属 zone
- 基础 tag
- 与主体的最小距离信息
- 是否靠边 / 靠路 / 靠主建筑

第一阶段不要让 AI 对整块剩余空间自由发挥，而是先让程序把“可补的位置”切成较清晰的次级可建造块。

## 6. 第一阶段落地方式

### 6.1 不预设固定默认类型

第一阶段不建议把次级建筑强行定死为“默认全部按 `single`”。

更合适的口径是：

- 不预设固定默认类型
- 但整体优先选择简单、局部、自洽、低扩展成本的落地形式
- 如果 AI 选择的是更复杂的扩展链，则需要满足更严格的可施工性条件

这样可以避免把第一阶段直接锁死为单一结构类型，同时仍保持“次级建筑整体降级”的方向。

### 6.2 复杂例外先不优先做

以下例外类型先不作为第一阶段重点：

- `pair`
- `linear`
- `start-expand`

这些类型可以先保留为后续增强项，而不是当前分支的阻塞目标。

## 7. 运行链路建议

建议第一阶段形成如下链路：

1. 主体建筑通过当前 `C7 -> C8 -> C9` 主链完成规划与落地
2. 主道路生成完成
3. 根据主体结果、主道路结果、留白区和 `secondary_fill` 生成次级可建造块
3. 对次级可建造块做基础过滤：
   - 是否满足面积要求
   - 是否离主体过近
   - 是否侵入主道路区域
   - 是否落入禁入区
   - 是否与其他次级块明显冲突
4. 为每个可建造块匹配通用候选模板池
5. 由 AI 在候选中挑选更适合作为次级结构的模板
6. 生成次级结构 placement
7. 将这些 placement 接入现有 `C9` queue / executor

这里建议明确一个基础公式：

`secondary_candidate_area = build_area - primary_module_rects - main_road_area - hard_reserved_area`

也就是说，次级可建造块不是在主体和道路之前预留，而是在它们稳定后，从剩余空间中切出。

## 8. 第一阶段最低规则建议

即使先做简化版，也建议至少定死以下规则：

### 8.1 主体避让

- 次级建筑不得压主体 footprint
- 次级建筑不得侵入主体主入口前方区域
- 次级建筑不得侵入已明确保留的核心留白区

### 8.2 次级之间的基本间距

- 次级建筑之间需要最小间距，避免过密或硬碰撞

### 8.3 边界与安全边距

- 次级建筑必须在可建造块内
- 对外边界需要保留内缩或安全边距

### 8.4 高度容差

- 次级建筑的地形容差建议低于主体建筑复杂链路的自由度
- 第一阶段优先选择更平整、更稳定的可建造块

### 8.5 exclusion zone 最小集合

为了避免引入大量参数，第一阶段建议只保留最小禁入集合：

1. 主体本体区
2. 主体入口区
3. 已明确保留的核心留白区

不建议第一阶段就引入：

- 多圈 buffer
- 多档权重
- 按建筑类型切不同 profile
- 复杂视线衰减或密度参数

## 9. AI 参与边界建议

第一阶段不建议让次级建筑也走完整的重 AI 语义规划。

建议边界如下：

- 程序负责生成次级可建造块
- 程序负责候选模板池与基础约束过滤
- 次级结构优先走轻量选择，而不是复杂工头式规划
- 仅当后续需要对重点次级建筑加语义时，再对少量块引入 AI

这能避免把刚稳定下来的主体主链再次复杂化。

## 10. 与道路系统的边界

当前道路更倾向外包，因此本案只需要保留最小接口意识：

- 次级可建造块或次级结构结果中，可以顺手保留“潜在道路接入点 / 朝向偏好 / 临边信息”
- 但本案不负责完整道路骨架生成
- 也不把道路规则作为当前分支的主阻塞项

## 11. 与现有代码的潜在映射

当前可优先关注的落点：

- `C6.secondary_fill` 作为输入起点
- 主模块与主道路稳定后生成次级可建造块
- `C9` 继续复用 placement / queue / executor

这意味着新分支更像是在现有主链旁补一条“次级建筑简化支线”，而不是推翻现有结构。

## 12. 当前最建议先拍板的问题

虽然已有不少结论，但在正式开工前，仍建议优先确认以下问题：

1. 主道路结果以什么结构提供给次级可建造块生成使用
2. 主体入口区与核心留白区在程序里各自怎么识别
3. AI 从通用候选中判断“哪些更适合作为次级结构”时，是否需要一条最小排除规则
4. 次级可建造块最终放在 `C8` 正式产物，还是放在更靠近 `C9` 的中间转换层

## 13. 推荐的第一阶段分支目标

如果要把本案转成实际开发目标，建议第一阶段只做：

- 从 `secondary_fill`、主体结果与主道路结果中生成次级可建造块
- 让 AI 在通用候选中挑选合适的次级结构
- 接入现有 `C9` 队列与执行器
- 做好主体避让、入口避让、核心留白避让、最小间距与边界限制

先把这条最小链路打通，再继续讨论复杂例外与语义增强。
