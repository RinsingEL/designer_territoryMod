# 核心建筑原型与排列方式驱动的横向拼图计划

## 计划摘要
本轮只做方案规划，不改代码。主流程收敛为：保留区域功能标签作为候选筛选入口，由 AI 长思考为 rect 选出更像“完整建筑本体或建筑群起点”的 `START / SINGLE`，并只决策 `building_archetype / building_name / starter_template / arrangement_type`。不再要求在这一步生成详细设计图或骨架图纸，实际落地一致性与合法性后移到 `C9` 判断。

## 实施步骤
1. 将主模块术语正式锁定为“核心建筑原型”。
2. 定义 rect 的最小稳定决策产物。
3. 明确 `START` 与 `SINGLE` 在横向主流程中的不同职责。
4. 明确 AI 长思考、排列算法、拼图机制、C9 审查层的职责边界。
5. 将整套流程整理为后续可落地的 skill 方案基础。

## 当前暂定工作流定义
1. 上游阶段先给每个 rect 保留区域功能标签，例如 `market / port / residential / defense`。
2. AI 按功能标签读取对应候选结构列表，不脱离功能语义自由乱选。
3. AI 重点审视候选中的 `START / SINGLE`，判断谁更像：
   - 完整建筑本体
   - 建筑群起点
4. 这一步明确标注为“需要 AI 长思考”的关键决策点，后续需要整合为 skill。
5. AI 为 rect 产出四个核心字段：
   - `building_name`
   - `starter_template`
   - `starter_pool`
   - `pool_chain`
   - `arrangement_type`
6. 若结果是 `SINGLE`：
   - 直接视为完整建筑本体
   - 不再追加横向排列决策
7. 若结果是 `START`：
   - 才需要 `arrangement_type`
   - 只决定排列方式，不在这一步生成详细设计图或拼图执行图纸
8. `C8` 未来按 `starter_template + arrangement_type` 驱动横向拼图扩张尝试。
9. `C9` 未来基于真实落地结果判断是否符合排列语义，必要时做拒绝、回退或标记。

## 当前补充定义
- `building_name` 由 AI 直接命名，只要能表达这整套建筑的语义即可。
- “建筑原型”的结构依据优先落在：
  - `starter_template`
  - `starter_pool`
  - `pool_chain`
- 也就是说，当前更看重 starter 和模板池链条，而不是先写死一个固定 `building_archetype` 枚举。
- `rect` 表示可支配建筑场域，不表示必须被建筑放满；留白可以自然作为庭院、花园、道路、缓冲带或次级建筑生成区。

## 模板池死循环约束
- 模板池链条的扩张必须受以下五层约束共同限制：
  1. 深度上限
  2. 已访问状态去重
  3. 禁止立即回边
  4. 面积/预算约束
  5. 终止件优先策略
- 实际扩张应理解为“在预算和排列方式约束下的受控遍历”，而不是沿模板池无上限递归追图。

## 设计取舍
- 放弃在决策阶段提前生成精细设计图。
- 原因是设计图需要与真实拼图连接行为严格对齐，调试成本高，而且容易与实际 jigsaw 落地脱节。
- 改为“先决定排列方式，后看真实扩张结果”，把一致性校验后移到 `C9`。

## 验收标准
- 已明确主模块、功能标签、starter、排列方式四者关系。
- 已明确 AI 长思考发生在 `START / SINGLE` 语义判断阶段。
- 已明确本轮只出方案，不改代码。
- 已明确不生成详细设计图，而是把一致性判断后移到 `C9`。
- 已明确模板池扩张必须受控，不能无约束递归。
- 已明确 `rect` 不必放满，留白是允许且合理的设计结果。

## 关键假设
- 当前最优先的是先把横向主流程和职责边界定死。
- 竖直方向目前视为已透传但未完整实现，不阻塞本轮规划。
- `C9` 被视为真实拼图结果的最终审查层，而不是单纯执行层。
