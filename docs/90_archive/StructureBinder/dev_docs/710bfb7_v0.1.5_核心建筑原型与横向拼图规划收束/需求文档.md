# 本次需求文档

## 用户需求摘要
新建一个当前任务目录，用来简略记录我们接下来要讨论的主模块语义、拼图机制和排列算法问题。

## 用户特别强调的约束
- 本轮只做方案规划，不改代码。
- 主模块统一定义为核心建筑原型。
- `START` 才需要 `arrangement_type`。
- 不在这一步生成详细设计图，把一致性判断后移到 `C9`。
- “判断哪个 `START / SINGLE` 更像完整建筑本体或建筑群起点”需要 AI 长思考，并且后续要沉淀为 skill。

## 当前已确定定义
- 主模块后续统一改称为“核心建筑原型”。
- 核心建筑原型不是 `market / port / residential` 这类区域功能标签。
- 核心建筑原型也不要求等于单个超大模板，它可以由一个 `START` 加若干拼图件逐步展开成完整建筑或建筑群。
- 当前更关注 starter 及其模板池链条，不优先写死固定 archetype 枚举。
- rect 的最小稳定决策产出固定为：
  - `building_name`
  - `starter_template`
  - `starter_pool`
  - `pool_chain`
  - `arrangement_type`

## 当前已确定工作流
1. rect 先拥有区域功能标签。
2. AI 按功能标签筛选候选结构列表。
3. AI 重点在候选中的 `START / SINGLE` 里判断“谁像完整建筑本体或建筑群起点”。
4. AI 给 rect 赋予具体建筑含义，并产出四个核心字段。
5. 若结果是 `SINGLE`，直接视为完整建筑本体。
6. 若结果是 `START`，才需要额外决定 `arrangement_type`。
7. `C8` 只负责按 `starter_template + arrangement_type` 驱动横向扩张。
8. `C9` 再基于真实落地结果判断是否符合排列语义。

## 当前已确定约束
- `building_name` 由 AI 命名，只要求语义清楚。
- `rect` 是可支配建筑场域，不是必须填满的建筑容器；允许保留留白给庭院、花园、道路、缓冲带或次级建筑。
- 模板池链条扩张必须至少受：
  - 深度上限
  - 状态去重
  - 禁止立即回边
  - 面积预算
  - 终止件优先
  这五层约束共同限制。

- 这次不实现 skill，只先把 skill 所需流程定清楚。
- 这次不设计详细建筑图纸或骨架图纸。
