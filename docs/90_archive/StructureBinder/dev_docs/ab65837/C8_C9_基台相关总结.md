# C8 / C9 基台相关总结

## 1. 这份文档的目的

这份文档用于总结当前项目里 `foundation` 相关字段的实际设计语义，避免后续继续实现 `C9` 时，把“历史上顺手留下的字段”误当成“必须重点建设的主系统”。

这里的结论以当前仓库中的代码和文档为准，优先描述“已经明确设计过什么”，其次才描述“更合理的推断”。

## 2. 一句话总结

当前这套设计里，`foundation` 更像是：

- `C8` 基于 build area 地形分析后，为 `C9` 提供的一份“基台/基础落位计划”

它的核心目标是回答这几个问题：

- 这块区域适合什么类型的基台
- 结构默认应该落在哪个基准高度
- 这块地形对应的基础范围大概在哪里
- 是否还有少量附带的基台补充信息

从当前语义强弱看：

- 核心字段：`foundation_type`、`base_y`
- 次核心字段：`terrain_impact_bbox`
- 可弱化字段：`supports`

## 3. foundation_type

`foundation_type` 是当前基台设计里的核心字段之一。

它表示 build area 对应的基台类型分类，当前程序会根据地形统计自动得出。现有规则大致是：

- 地势很平：`NONE`
- 起伏非常大：`TERRACE`
- 边缘高差明显且起伏较大：`PLATFORM_WITH_RETAINING_WALL`
- 其他情况：`PLATFORM`

因此它本质上是：

- 对地形处理方式的粗分类
- 对基台样式的高层语义归纳

它不是逐块施工蓝图，也不是单个 piece 的精细落地规则。

## 4. base_y

`base_y` 是当前基台设计里的另一个核心字段。

它表示该 build area 的基础参考高度，也是 `placements` 默认落位时会对齐的基准高度。

从当前代码看：

- `C8` 会先根据地形统计求出 `base_y`
- 然后把 `placements` 中各节点的 `y` 默认写成这个值

因此它的作用可以理解为：

- 基台的参考施工高度
- 结构默认落地的统一基准面

如果后续 `C9` 只保留最小可用实现，那么 `base_y` 仍然是必须保留的字段。

## 5. supports

`supports` 是一个轻量级附带字段，不像 `foundation_type` 和 `base_y` 那样核心。

它的结构非常简单，每一项只有：

- `type`
- `side`

当前项目里，文档和代码都表明它主要用于表达少量“基台附带构件”信息。例如：

- `retaining_wall`
- `stairs`

而且这些条目并不是 AI 单独决定的，而是程序根据 `foundation_type` 和边缘地形情况自动补出的。

更贴近当前项目语义的理解是：

- `supports` 不是一个完整的支撑物系统
- 它更像“基台方案里可能顺手带出来的附带构件提示”

也就是说，它存在的原因很可能只是：

- 某些基台方案本身带挡墙
- 某些基台方案本身带台阶
- 所以 `C8` 顺手把这类附属信息也挂到了 foundation 上

从当前实现成熟度看，`supports` 并没有发展成独立主系统，因此后续如果继续做 `C9`：

- 可以消费它，但不建议把它当成核心设计轴心
- 也可以在 foundation 简化后弱化它的存在感

## 6. terrain_impact_bbox

`terrain_impact_bbox` 是一个 area 级别的矩形边界框，包含：

- `minX`
- `minZ`
- `maxX`
- `maxZ`

当前 `C8` 会根据 build area 的地形分析结果生成这个 bbox，并向外扩一圈。

目前能确定的是：

- 它是 foundation 的正式输出字段之一
- 它不是单个 piece 的坐标
- 它也不是精确到 block 的施工列表

更稳妥的理解是：

- 它表示 foundation 相关的一块 terrain 范围框
- 供 `C9` 或调试/复核阶段参考

需要注意的是：

- 现在能确认它是“foundation 相关 terrain bbox”
- 但不能过度下结论，说它已经被明确固定为“施工影响范围”或“回滚范围”

所以后续使用它时，建议保持宽口径，不要先入为主地把它绑定到某一种唯一用途。

## 7. 当前更合理的整体理解

如果把这些字段合起来理解，当前 foundation 相关数据更像：

- `foundation_type`：这块地该用哪类基台
- `base_y`：这块地的基台/结构参考高度
- `terrain_impact_bbox`：这块 foundation 相关的 terrain 边界框
- `supports`：少量附带构件提示

因此 foundation 的主干语义主要还是：

- 类型
- 高度

而不是：

- 复杂支撑系统
- 完整地形施工 DSL

## 8. 对 C9 实现的建议口径

如果后续继续推进 `C9`，建议按下面的优先级理解这些字段：

### 第一优先级

- 消费 `foundation_type`
- 消费 `base_y`

这是 foundation 最核心、最稳定的两部分。

### 第二优先级

- 视需要使用 `terrain_impact_bbox`

它更适合作为：

- foundation 相关范围提示
- 调试/校验/报告辅助信息

### 第三优先级

- `supports` 可选消费

如果未来确实要恢复“某类基台自动补挡墙或台阶”的效果，可以使用它；如果不做这层细化，它也不是当前 `C9` 必须先打通的主线。

## 9. 最终结论

可以把当前项目里的 foundation 设计总结为：

- 它本质上是一份“基台相关计划”
- 其中真正稳定且核心的是 `foundation_type` 和 `base_y`
- `terrain_impact_bbox` 是有效的范围信息，但语义不要过度收窄
- `supports` 更像历史上为基台附带构件预留的小出口，不必过度扩展成大系统

如果后续重构 `C9`，优先围绕“基台类型 + 基准高度”来建设，会比优先围绕 `supports` 更稳。
