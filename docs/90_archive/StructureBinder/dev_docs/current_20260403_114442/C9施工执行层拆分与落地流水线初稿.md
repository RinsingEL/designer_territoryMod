# C9 施工执行层拆分与落地流水线初稿

## 1. 背景

当前 `C9` 主链已经基本统一为：

- `C8` 产出已校验的结构节点
- `C9` 将可执行节点转成 `BuildQueue`
- `CityBuildQueueExecutor` 在运行时按队列执行
- 实际结构放置最终调用 `StructureInjector.spawnStructureAtBlock(...)`

这条链路作为当前主模块落地基线已经可用，但执行层内部的职责仍较集中，尤其是：

- 队列管理
- 区块加载门控
- 运行时校验
- 结构实际放置
- 地形预处理 / 原世界清理

这些内容继续堆在 `CityBuildQueueExecutor`、`CityC9Stages` 和 `StructureInjector` 内，会让后续次级建筑、道路附属结构、嵌入式结构等需求越来越难接。

因此，本案目标不是推翻现有主链，而是把它拆成更明确的执行流水线。

## 2. 当前主链现状

### 2.1 已经成立的稳定主链

当前真正运行的主链是：

1. `CityController.handleCityC9Generate(...)`
2. `CityC9Stages.generate(...)`
3. `CityC9BuildQueue.upsertFromPlan(...)`
4. `CityBuildQueueExecutor.executeLoadedTasksNow(...) / executeReady(...) / executeTask(...)`
5. `StructureInjector.spawnStructureAtBlock(...)`

这条链已经支持：

- 先预存建造计划
- 区块加载后进入可执行状态
- 运行时校验父节点与碰撞
- 真正把结构写入世界

### 2.2 当前遗留问题

虽然主链已经统一，但仓库中仍存在一些不适合继续扩张的情况：

- `CityC9Stages` 里仍保留了旧式直落辅助逻辑
- 结构放置和运行时校验仍过度耦合
- 地表清理、嵌入式挖空等需求还没有明确落点
- 高度口径目前分散在不同类和不同调用场景中

## 3. 新方案的核心目标

本案希望把当前执行层收成一条更明确的流水线：

1. 队列管理
2. 区块门控
3. 预检查
4. 地形预处理
5. 结构放置
6. 后处理
7. 运行时结果记录与失败处理

也就是说，后续不再把“放结构”理解成一个孤立方法，而是一整段可扩展的施工过程。

## 4. 拆分原则

### 4.1 保留的编排层

以下类建议继续保留编排职责，不急着拆碎：

- `CityController`
- `CityC9Stages`
- `CityBuildQueueExecutor`

它们更适合负责：

- 接口入口
- 阶段编排
- 执行调度

### 4.2 建议拆出的执行能力

以下内容更适合独立为可复用能力层：

- 队列管理
- 区块门控
- 地形预处理
- 结构落地
- 运行时校验
- fallback / repair

## 5. 建议拆分的服务层

### 5.1 `BuildQueueManager`

职责：

- 管理 `BuildQueue / BuildTask`
- 队列落盘与加载
- 任务索引与状态流转
- 按城市 / group / chunk 维护任务映射

可从当前代码中吸收的内容：

- `CityC9BuildQueue`
- `CityBuildQueueExecutor` 中与索引、状态、持久化直接相关的部分

### 5.2 `BuildChunkGate`

职责：

- 监听区块加载事件
- 判断区块是否达到可施工状态
- 决定任务从 `PLANNED` 何时转入 `READY`
- 维持“先预存计划，后延迟落地”的执行模式

这层的存在意义是明确表达一个事实：

当前 mod 的目标不是“当场立即落地一切”，而是“早期保存计划，等区块真正可施工时再执行”。

### 5.3 `BuildTerrainPreparationService`

职责：

- 落地前清理地表障碍
- 清树叶、树干、灌木、零散 feature 方块
- 嵌入式结构挖空或清腔
- 必要时做简单整地
- 记录清理范围，必要时支持回滚

这一层非常关键，因为后续无论是：

- 普通房屋落地前清表
- 次级建筑避开或清理地物
- 嵌入洞中的结构
- 半埋式地基

都不应该继续直接塞进 `spawnStructureAtBlock(...)`。

### 5.4 `BuildPlacementService`

职责：

- 封装真实结构放置入口
- 统一调用 `StructureInjector.spawnStructureAtBlock(...)`
- 处理 placement bounds / template size / snapshot / restore
- 统一对外提供“按任务或按 placement 放结构”的接口

这层可以理解为“落地方法封存”的正式落点。

### 5.5 `BuildRuntimeValidator`

职责：

- 父节点依赖检查
- 运行时 footprint collision
- placement bounds 合法性
- 与已完成任务的冲突判断
- 后续可继续接入次级结构、道路接口等运行时限制

它和 `TerrainPreparation` 的区别是：

- `TerrainPreparation` 处理“落之前要改世界什么”
- `RuntimeValidator` 处理“这次施工是否允许发生”

### 5.6 `BuildFallbackService`

职责：

- 失败后的 retry / blocked 决策
- terminalize / replace / restore 等 fallback
- 后续更复杂的 repair 策略入口

当前这层可以先保持较轻，但建议早点独立边界，避免以后继续散在 `C9Stages` 与 `Executor` 里。

## 6. 建议的最小施工流水线

建议一个任务真正执行时，按下面的顺序推进：

1. `PRECHECK`
   - 区块是否 FULL
   - 父节点是否已完成
   - 当前 placement bounds 是否可计算

2. `PREPARE_TERRAIN`
   - 清理地表障碍
   - 清 feature
   - 必要时挖空或填方
   - 记录清理上下文

3. `PLACE_STRUCTURE`
   - 真正执行结构放置

4. `POST_CLEANUP`
   - 清 jigsaw
   - 清临时残留块
   - 补简单收尾

5. `POST_VALIDATE`
   - 确认最终状态
   - 若失败则进入 retry / blocked / fallback

这样可以避免未来再把“清地表”“嵌入洞里”“普通摆放”都写进一个单函数里。

## 7. 关于高度口径的建议

### 7.1 当前已出现的口径

当前仓库中已经能看到至少三类高度口径：

- `WORLD_SURFACE`
- `MOTION_BLOCKING_NO_LEAVES`
- `OCEAN_FLOOR_WG`

并且 `StructureInjector` 中已经明确写出：

- `OCEAN_FLOOR_WG` 更接近固体地表，忽略树木和水
- `WORLD_SURFACE` 会拿到包含树叶的最高点

### 7.2 本案建议

后续不建议继续在业务代码中零散调用 `level.getHeight(...)` 或 `generator.getBaseHeight(...)`。

更建议新增一层统一解析器，例如：

- `BuildSurfaceResolver`
- 或 `TerrainHeightResolver`

统一提供几种明确口径：

- `LIVE_WORLD_SURFACE`
- `LIVE_SURFACE_NO_LEAVES`
- `GEN_BASE_OCEAN_FLOOR`
- `GEN_BASE_WORLD_SURFACE`

这样每个施工步骤都能显式说明自己使用哪种高度定义。

### 7.3 对当前问题的结论

如果目标是：

- 找当前可施工地表
- 不希望树叶抬高地面

那么 `MOTION_BLOCKING_NO_LEAVES` 比 `WORLD_SURFACE` 更合适。

但如果目标是：

- 获取更接近原始地形基底的高度
- 尽量不受运行时 feature 和后续世界变更影响

则应优先考虑 `generator.getBaseHeight(...)` 一类口径，而不是单纯依赖当前世界的 `level.getHeight(...)`。

## 8. 关于区块加载前预存计划

当前项目的执行思路应明确保留：

- 建造计划可以在区块加载前就持久化
- 区块真正进入可施工状态后，再从队列执行落地

这套思路的好处是：

- 结构规划与区块是否已加载解耦
- 更适合大范围预埋城市施工计划
- 便于后续在玩家接近区域时逐步兑现

因此，不建议后续重构时把执行层改回“生成后立即同步放置”的模式。

## 9. 与后续次级建筑的关系

本案并不只服务主体建筑。

如果后续要做次级建筑、道路附属结构、入口标识、广场装饰等内容，它们都应该尽量复用同一套执行层：

- 共用队列系统
- 共用 chunk gate
- 共用 terrain prepare
- 共用 placement service
- 共用 runtime validator

这样才能避免后续每加一类结构，就又长出一套新的落地器。

## 10. 当前最建议先拍板的问题

1. 地形预处理第一阶段最小支持哪些动作：
   - 清树叶
   - 清树干
   - 清灌木
   - 清单格地物
   - 挖空
   - 填平

2. `BuildSurfaceResolver` 是否现在就值得立项，而不是继续散写高度口径

3. 旧 `CityC9Stages` 中未进入主流程的直落逻辑，是否在执行层重构时一起清理

4. fallback 未来是否只保留在主体建筑，还是也要面向后续次级建筑开放统一入口

## 11. 推荐的第一阶段重构目标

如果把本案转成实际开发，建议第一阶段只做：

- 明确 `BuildQueueManager`
- 抽出 `BuildPlacementService`
- 抽出 `BuildRuntimeValidator`
- 预留 `BuildTerrainPreparationService` 的正式入口
- 保留当前“预存计划 -> 区块 FULL 后执行”的整体模式

这样可以先把主结构收干净，再逐步接入更复杂的地形清理与嵌入式施工。
