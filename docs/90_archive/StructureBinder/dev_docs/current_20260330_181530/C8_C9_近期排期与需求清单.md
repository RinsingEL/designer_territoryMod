## 任务 2A：C8 排列方式求解器设计收口

### 当前状态

- 当前 `C8` 仍主要按 `arrangement_type` 驱动排列求解。
- 现有四类排列方式只适合作为近期最小示例集合，不足以覆盖更多主体建筑类型。
- 已讨论确认：巴洛克教堂、罗马神殿、四合院、`U` 型港等结构，难以被固定排列枚举稳定覆盖。
- 因此，后续方向应从“排列方式枚举”升级为“排列方式求解器 + 约束”的统一模型。

### 已达成方向

- 现有四类排列方式继续保留，但降级为首批示例预设，不再视为最终通用分类：
  - 轴对称展开
  - 沿主干线两侧展开
  - 环状展开
  - 从圆心向外展开
- `C8` 后续不再只依赖单个 `arrangement_type` 做硬分支，而是逐步升级为按求解器描述求 `placement`。
- 求解器不直接由建筑名词驱动，而应由空间语义、拓扑偏好、约束规则和模板实例共同驱动。
- `C7` 负责给出求解描述，`C8` 负责求解，`C9` 负责运行时校验与 fallback。

### 当前拟定的求解器分层

- 语义层：只保留稳定空间语义，例如：
  - `anchor`
  - `axis`
  - `wing`
  - `void`
  - `edge`
  - `corner`
  - `terminal`
  - `junction`
  - `gate`
  - `core`
- 拓扑层：表达结构骨架关系，不直接使用建筑类型名，例如：
  - 串联
  - 围合
  - 中心核
  - 辐射
  - 群组
  - 环
  - 分叉
- 约束层：承载真正易变化的规则，例如：
  - 是否必须有主轴
  - 是否必须保留 `void`
  - 是否允许闭合
  - 是否允许开口 `U` 形
  - 前后锚点最小距离
  - 左右是否要求严格镜像
  - 是否允许局部非对称修正
  - 是否优先形成连续边
  - 是否允许单边先行再补另一边
- 模板 / 实例层：只提供具体候选组件、权重、尺寸、风格标签和连接器能力，不直接决定求解器本体。

### 待策划问答

- [x] Q1：主体建筑在本轮的最低通过标准是什么？
  已回答：
  - 当前阶段不要求最终完整成城，但必须满足以下最低标准：
    - 日志中能明确表现出节点生成过程是可控的
    - 选出来的结构符合当前排列算法
    - 每个待落地节点在落地前都经过地形检查

- [x] Q2：如果核心节点之后的部分失败，本轮是否接受“保留已成立父链并 terminalize 收尾”作为通过结果？还是只要主链被截断就必须判失败并 blocked？
  已回答：
  - 当前阶段接受 terminalize / 收尾替换作为有效 fallback。
  - 但 fallback 不建议等到节点已经正式 place 后再整体回退，而是应在正式落地前先预判一层后继是否还能合法继续生成。
  - 如果预判发现后续不能合法继续，而当前节点本身又不是收尾类型，则不进入正式放置，直接改为收尾类型后再落位。
  - 如果当前节点本身就是收尾类型，则直接保留。
  - 当前阶段优先采用“place 前预判一层，不合法则直接收尾替换”的口径，避免先放再回退带来的额外处理成本。
