# 8a2497f - Improve city template selection debugging and market catalog

## 背景

本轮围绕城市链路 `C6 -> C7 -> C8 -> C9` 做了连续测试，目标是让 `market` 类型结构在当前测试版辅助数据下能够稳定进入自动选模与落地流程。

期间重点测试对象为：

- `city_3120_-3200`
- `g_market_04`
- `g_market_03`

## 本次遇到的问题

### 1. `market` 候选过少，严格模式下经常为 0

最初 `C7` 在严格模式下无法为 `market` 组选到模板，原因主要有两类：

- 现有 catalog 里 `market` 条目太少
- 一批看起来可用的 `shop/commercial` 结构虽然格式正确，但来源仍被记成 `mcp_path_heuristic`，会被严格模式过滤掉

### 2. `connector` 语义混杂

之前系统里存在以下混用情况：

- 真实 `connectors`
- `connector_dirs`
- `orientation.jigsaw_facing`

这会导致：

- 一些没有真实水平 jigsaw 连接点的模板被误当成“可扩展拼图件”
- `C7/C8` 语义不够稳定

### 3. `C9` 队列会被旧的 `DONE` 任务卡住

同一个 `task_id` 对应的模板、坐标已经变化时，队列仍然把旧任务视为“已完成”，导致新任务无法重新入池。

### 4. `g_market_03` 的 `C6` 模块尺寸分桶错误

`C6` 在把矩形写成 `primary_modules` 时，`template_hint.size_tier` 是按**整个功能区面积**算的，而不是按每个矩形自身面积算的。

结果是：

- `g_market_03` 明明切了 3 个中型矩形
- 但 3 个主模块都被错误地标成了 `L`
- 后续 `C7` 找不到合适的 `market` 模板

### 5. `C8` 现场链路仍存在刷新/写回异常

在 `g_market_03` 上，`C7` 已经能够选到新的 `shop -> market` 模板，但 `C8` 现场返回与 group 文件内容之间出现过不一致现象，因此补充了额外日志用于继续排查。

## 本次提出的需求

本轮明确形成并落地了以下需求：

1. `connector` 语义要收紧为“结构 NBT 内真实存在的水平 jigsaw 连接点”
2. 无 connector 模板应视为合法单体终止件，不应被误判为必须扩展
3. `C7` 需要支持严格/宽松两种 tag source 模式
4. 严格模式默认保留，但应支持全局开关与单次请求覆盖
5. `shop` 类结构需要作为可信 `market` 候选参与自动选模
6. 排查链路要增加高信号日志，避免反复纠结“当前运行的到底是哪条逻辑”

## 本次做出的修改

### A. `C9` 队列更新逻辑

新增并使用 `placement_signature`，使队列在以下情况下能够正确重入：

- 同一 `task_id`
- 但模板、坐标、rotation、footprint 等内容变化

行为调整为：

- 内容不变且 `DONE`：跳过
- 内容变化：覆盖旧任务并重置为 `PLANNED`

### B. `connector` 语义收紧

在 `TerraSense` 与 `StructureBinder` 两边统一调整：

- `connectors` 作为唯一拼图连接真源
- `connector_dirs` 退化为摘要字段
- `orientation.jigsaw_facing` 仅保留扫描事实
- `C7/C8` 不再把 `connector_dirs/jigsaw_facing` 当成 connector fallback

### C. `market` 候选补强

将以下结构提升为可信 `market` 候选：

- `minecraft:village/plains/houses/plains_butcher_shop_1`
- `minecraft:village/plains/houses/plains_butcher_shop_2`
- `minecraft:village/savanna/houses/savanna_butchers_shop_1`
- `minecraft:village/taiga/houses/taiga_butcher_shop_1`

调整内容包括：

- `primary_function -> market`
- `function_candidates -> market + commercial`
- `preset_pool -> village_market_small`
- `source/tag_rule -> manual_temp_override`
- `manual_override -> true`

### D. 严格模式控制开关

新增全局配置文件：

- `config/terra_script_city_generation.json`

新增配置读取类：

- `CityGenerationConfig`

行为改成：

- 全局默认由 `strict_tag_source_default` 控制
- `/structure_templates_query` 与 `/city_c7_generate` 可读取该默认值
- 请求体中显式传 `strict_tag_source` 时可覆盖全局默认

### E. `C6` 主模块尺寸分桶修复

修复 `C6` 在 `applyAcceptedDecision(...)` 中对 `template_hint.size_tier` 的赋值方式：

- 旧逻辑：按整个 build area 的 `area_blocks`
- 新逻辑：优先按单个 rect 的 `w * h`

这使得大功能区中的中等矩形不会再被整体误分成 `L`

### F. 调试日志增强

补充了 `C6/C8` 的高信号日志，用于现场确认：

- `C6` 当前按哪个面积在计算 `size_tier`
- `C8` 实际读到了哪个 `C7 selected_template`
- `C8` foundation 生成和 group 文件写回是否一致

## 当前结果

### 已验证通过

- `C9` changed-task 覆盖重入逻辑
- 严格 `market` 查询与 `shop -> market` 候选放行
- `g_market_04` 的 `C7` 自动选模可使用新的 market/shop 候选
- `g_market_03` 的 `C6` 可通过矩形校验并写入布局
- `g_market_03` 的 `C7` 在修复 `size_tier` 后能够选到 `plains_butcher_shop_1`

### 仍在继续排查

- `g_market_03` 的 `C8` 现场链路存在“返回成功，但 group 级 foundation 文件内容与预期不一致”的现象
- 为此已补充日志，后续应基于新的 `[C8] ...` 日志继续定位是读取、生成还是写回阶段的问题
