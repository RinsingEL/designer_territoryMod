# T1 国度和区域蓝图生成

## 当前状态

- 状态：部分实现
- 方案来源：`docs/solution.md`
- 当前代码已经落地了三段里的两段：
  - 程序候选区扫描与选簇取点
  - 蓝图落盘与校验
- 真正的“AI 编排器”没有独立做成 `T1` stage/controller，而是由外部工作流把 `WorldController` 和 `TerritoryController` 串起来

## T1 在当前项目里的真实分层

虽然 `solution.md` 里把 T1 写成一个完整阶段，但当前代码里它实际拆成三段：

1. `Q1` 候选区域扫描与预览
2. `Q2` 从候选簇中选一个点
3. 提交最终蓝图并落盘

也就是说，T1 目前不是一个单独的 Java 阶段类，而是“世界查询接口 + 蓝图仓库”的组合流程。

## 一、Q1：候选区域扫描与预览

### 入口

| 类 | 方法 | 说明 |
| --- | --- | --- |
| `server.mcp.WorldController` | `handleQueryRegion(HttpExchange)` | 对 region 或 territory 生成候选区、ASCII 图、可视化预览 |
| `server.mcp.WorldController` | `handleT1PreviewMaps(HttpExchange)` | 对指定 `region_id` 输出 T1 预览图 |

### 核心类与关键方法

| 类 | 方法 | 作用 |
| --- | --- | --- |
| `server.mcp.WorldController` | `handleQueryRegion(...)` | 组织整条 Q1 逻辑 |
| `server.mcp.WorldController` | `parseCriteria(...)` | 解析过滤条件 |
| `server.mcp.WorldController` | `filterCandidates(...)` | 按 slope/TPI 过滤基础像素 |
| `util.DBSCAN` | `cluster(...)` | 把候选点聚成簇 |
| `server.mcp.WorldController` | `buildClusterJson(...)` | 计算单簇描述、关键点、ASCII 图 |
| `domain.world.stage.QueryRegionPreviewExporter` | `export(...)` | 输出候选簇覆盖在底图上的预览图 |
| `server.mcp.WorldController` | `persistPendingSelection(...)` | 缓存待选结果，供 Q2 使用 |
| `domain.world.stage.T1PreviewExporter` | `export(...)` | 输出指定 region 的地貌预览图 |

### 程序具体怎么计算候选区

#### 1. 确定扫描范围

`handleQueryRegion(...)` 支持两种目标：

- `region_id`
- `territory_id`

分支逻辑：

- 如果传 `territory_id`
  - 从 `TerritoryManager.getAllResults()` 找目标领地
  - 用 `TerritoryManager.globalOwnershipMap` 和 `ScanResultHolder.lastScanData` 提取该领地归属范围内的 land pixel
- 如果传 `region_id`
  - 从 `ScanResultHolder.regionCacheMap` 取细扫缓存
  - 读取 `detailData`、`slopeData`、`tpiData`
  - 采样出该 region 内的 land pixel

#### 2. 按条件过滤候选像素

`parseCriteria(...)` 会解析用户或 AI 传入的约束：

- `min_slope`
- `max_slope`
- `min_tpi`
- `max_tpi`

`filterCandidates(...)` 的逻辑：

- 先跳过 `null` 或非 land 点
- 如果是 `region_id` 分支
  - 用 `sample(cache.slopeData, ...)`
  - 用 `sample(cache.tpiData, ...)`
  - 判断是否落在阈值区间内
- 如果是 `territory_id` 分支
  - 目前 `checkCriteria(...)` 还是占位实现，直接返回 `true`
  - 也就是说 territory 分支当前主要按“归属范围 + land”筛点，尚未做和 region 分支等价的地形条件过滤

#### 3. 聚类形成候选簇

`DBSCAN.cluster(filtered, step * 4.0, 5)` 会把过滤后的点做 DBSCAN 聚类。

关键参数含义：

- 聚类半径：`step * 4.0`
- 最小点数：`5`

聚类结果会按簇大小从大到小排序，默认取前 `limit` 个。

如果传了 `interest_groups[]`，程序会按组分别过滤、分别聚类、分别限流，最后给每组生成：

- `group_id`
- `label`
- `symbol`
- `preview_label`

#### 4. 生成每个候选簇的描述

`buildClusterJson(...)` 做了三件事：

- 调 `AsciiMapGenerator.generate(...)` 生成 ASCII 小地图
- 计算平均 `slope` 和平均 `tpi`
- 用规则生成一个人可读描述 `description`

描述规则：

- `avgTpi > 1.0`：`Ridge/Highland`
- `avgTpi < -1.0`：`Valley/Basin`
- 否则：`Plain`
- 再根据 `avgSlope` 追加：
  - `> 1.5`：`(Rugged)`
  - `< 0.5`：`(Flat)`

同时写出：

- `metrics.size`
- `metrics.biome`
- `key_points`
- `ascii_map`

#### 5. 输出预览图并挂起待选结果

`QueryRegionPreviewExporter.export(...)` 会：

- 先用底层 `mapData` 渲染连续地形背景
- 再把候选簇叠加为彩色 overlay
- 给每个簇画 label
- 输出预览图和 legend

`persistPendingSelection(...)` 会把本轮候选缓存到：

- `terra_script/cache/query_region/query_region_selection_<queryId>.json`
- `query_region_selection_latest_<target>.json`

这样 Q2 不需要重新跑一遍聚类。

### Q1 输入

- `region_id` 或 `territory_id`
- 可选 `min_slope/max_slope/min_tpi/max_tpi`
- 可选 `limit`
- 可选 `limit_per_group`
- 可选 `interest_groups[]`

### Q1 输出

- `step = Q1_SCAN_PREVIEW_PENDING_PICK`
- `candidates[]`
- `candidates_metadata[]`
- `group_ascii_maps[]`
- `visual_map`
- `preview_overlay`
- `selection_pending.query_id`

### T1 预览图接口

`handleT1PreviewMaps(...)` 用于对单个 `region_id` 生成地貌预览图。

核心方法是 `T1PreviewExporter.export(...)`，主要逻辑：

- 解析 region 对应的地图数据
- 用 `TerrainFeatureComputer.computeSlope(...)` 计算坡度
- 下采样得到：
  - 高程图
  - hillshade 图
  - slope 图
  - biome 图
- 输出 PNG 和对应 legend

这里属于纯程序计算，不含 AI 决策。

## 二、Q2：从候选簇中选点

### 入口

| 类 | 方法 | 说明 |
| --- | --- | --- |
| `server.mcp.WorldController` | `handleQueryRegionPick(HttpExchange)` | 从 Q1 的 pending selection 里选簇和选点 |

### 核心类与关键方法

| 类 | 方法 | 作用 |
| --- | --- | --- |
| `server.mcp.WorldController` | `loadPendingSelection(...)` | 读取 Q1 缓存 |
| `server.mcp.WorldController` | `pickCandidate(...)` | 按 `cluster_id/label/preview_label` 选簇 |
| `server.mcp.WorldController` | `resolvePointFromMode(...)` | 选出最终点位 |
| `server.mcp.WorldController` | `persistPickResult(...)` | 保存 Q2 结果 |

### 程序具体怎么选点

#### 1. 先定位目标簇

`pickCandidate(...)` 支持三种选簇方式：

- `cluster_id`
- `label`
- `preview_label`

按顺序匹配，匹配到第一个就返回。

#### 2. 再从簇的关键点里选最终点

`resolvePointFromMode(...)` 支持：

- `center`
- `north`
- `south`
- `east`
- `west`
- `random_cardinal`

内部映射关系：

- `north -> north_tip`
- `south -> south_tip`
- `east -> east_tip`
- `west -> west_tip`
- `center -> center`
- `random_cardinal`：在四个 cardinal tip 里随机抽一个；如果抽不到就回退到 `center`

#### 3. 落盘 Q2 结果

`persistPickResult(...)` 会把结果写入：

- `terra_script/cache/query_region/query_region_pick_<queryId>_c<clusterId>.json`

### Q2 输入

- `query_id`
- 或 `target_type + target_id`
- `cluster_id` / `label` / `preview_label`
- `point_mode`

### Q2 输出

- `step = Q2_PICK_CLUSTER_POINT`
- `selected_cluster`
- `selected_point`
- `point_mode_applied`
- `from_pending.preview_overlay`

## 三、蓝图提交与落盘

### 入口

| 类 | 方法 | 说明 |
| --- | --- | --- |
| `server.mcp.TerritoryController` | `handleT1Blueprint(HttpExchange, MinecraftServer)` | GET/POST 蓝图 |

### 核心类与关键方法

| 类 | 方法 | 作用 |
| --- | --- | --- |
| `territory.io.TerritoryRepository` | `saveBlueprint(...)` | 校验并保存蓝图 |
| `territory.io.TerritoryRepository` | `listBlueprints(...)` | 读取全部蓝图 |
| `territory.io.TerritoryRepository` | `loadBlueprints(...)` | 从文件读取列表 |
| `territory.io.TerritoryRepository` | `upsert(...)` | 按 `territory_id` 覆盖或新增 |
| `territory.io.TerritoryRepository` | `validate(...)` | 校验蓝图字段 |

### 程序如何校验和保存

`saveBlueprint(...)` 的具体实现顺序：

1. 判空：`server`、`payload`
2. 用 Gson 把 `payload` 反序列化成 `TerritoryBlueprint`
3. 调 `validate(...)` 做字段校验
4. 读取现有蓝图列表 `loadBlueprints(...)`
5. 调 `upsert(...)`：
   - 若 `territory_id` 已存在，则覆盖
   - 否则追加
6. 写回 `T1_Blueprint.json`

`validate(...)` 当前硬校验字段：

- `territory_id`
- `name`
- `target_continent_id > 0`
- `expansion_policy != null`
- `expansion_policy.base_power > 0`
- `expansion_policy.costs != null`

### 蓝图数据结构

`TerritoryBlueprint` 当前字段：

- `territory_id`
- `name`
- `target_continent_id`
- `narrative`
  - `theme`
  - `description`
  - `ruler`
  - `color`
- `expansion_policy`
  - `base_power`
  - `costs`
    - `base_move`
    - `slope_penalty`
    - `water_penalty`
    - `forest_penalty`
    - `preferred_biomes[]`
    - `avoid_biomes[]`

### 输入

- POST 蓝图 JSON
- GET 无输入

### 输出

- `step = T1`
- `ok`
- `message`
- `blueprint` 或 `blueprints[]`

## 四、AI 决策和程序计算的边界

这是 T1 最重要的部分，当前代码里建议按下面理解。

### 1. 程序计算的部分

- 从 world / region / territory 数据里提取基础像素范围
- 计算 slope / tpi 条件过滤
- 做 DBSCAN 聚类
- 计算每个候选簇的：
  - 面积
  - 平均坡度
  - 平均 TPI
  - biome
  - 关键点
  - ASCII 图
  - 预览图 overlay
- 缓存 pending selection
- 校验蓝图 JSON 是否合规
- 保存蓝图列表

这些都是“可重复、确定性、程序型”的工作。

### 2. AI 决策的部分

- 决定关注哪些候选区
- 决定 `interest_groups` 应该长什么样
- 决定最终选择哪个候选簇
- 决定点位模式：
  - `center`
  - `north`
  - `south`
  - `east`
  - `west`
  - `random_cardinal`
- 决定国家叙事：
  - `theme`
  - `description`
  - `ruler`
  - `color`
- 决定扩张策略：
  - `base_power`
  - 各类移动 cost / penalty
  - `preferred_biomes`
  - `avoid_biomes`

这些都是“策略、偏好、叙事、目标函数”的部分，应该由 AI 或人工给出，而不是程序写死。

### 3. 当前还没完全打通的地方

- 目前没有单独的 `T1` orchestration 类去统一驱动 Q1、Q2、蓝图提交。
- `handleQueryRegion(...)` 里的 territory 分支尚未真正按 slope/TPI 做过滤，`checkCriteria(...)` 还是占位实现。
- T1 现在更像“由外部工作流编排的多接口流程”，不是一个单体阶段。

## 五、方案文档中的目标语义

- 不只做国家设定，还要做“多候选战略区筛选”。
- 流程应与 T2 一致：
  - 先扫描多个感兴趣区域并生成预览挂起（Q1）
  - 再人工触发选簇取点（Q2）
  - 最后提交蓝图

## 六、当前代码与方案的映射

- Q1 对应：`WorldController.handleQueryRegion(...)`
- Q2 对应：`WorldController.handleQueryRegionPick(...)`
- T1 预览图对应：`WorldController.handleT1PreviewMaps(...)`
- 蓝图落盘对应：`TerritoryController.handleT1Blueprint(...)`
- 当前缺的不是“数据结构”，而是“把这几步正式定义成一个完整 T1 工作流阶段”的编排层
