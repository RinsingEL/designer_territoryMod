# C1 选址与城市参数

## 当前状态

- 状态：规划中，未按 `solution.md` 完整落地
- 当前代码里最接近 C1 的入口是：
  - `CityController.handleCreateCity(...)`
  - `T4Stage.buildCityBootstrapConfig(...)`

## 阶段目标

- 把“城市建在哪里、为什么选这里、层级怎么配置”标准化
- 理想上应采用和 T1/T2 类似的候选 -> 选择 -> 固化流程

## 一、当前代码中的映射入口

| 类 | 方法 | 说明 |
| --- | --- | --- |
| `server.mcp.CityController` | `handleCreateCity(HttpExchange)` | 直接接收 `CityConfig` 并创建城市 |
| `domain.territory.stage.T4Stage` | `buildCityBootstrapConfig(...)` | 从 territory 首都自动派生 CityConfig |

## 二、核心类与关键方法

| 类 | 方法 | 作用 |
| --- | --- | --- |
| `world.city.CityConfig` | `resolveLayerLayout()` | 规范化层配置 |
| `server.mcp.CityController` | `handleCreateCity(...)` | 解析请求并校验中心点 |
| `server.mcp.CityController` | `probeCenterWater(...)` | 判断中心点是否落在重水域 |
| `world.city.CityManager` | `createCity(...)` | 根据配置生成城市 |

## 三、程序当前具体做了哪些校验

### 1. 基础字段校验

至少要求：

- `territoryId` 不为空

### 2. 中心点水域校验

当 `allowWaterCity = false` 时，`handleCreateCity(...)` 会调用 `probeCenterWater(...)`：

- 半径：`8`
- 步长：`2`
- 统计周边水体比例

拒绝条件：

- 中心点本身是水
- 或 `water_ratio >= 0.60`

### 3. 层配置标准化

`CityConfig.resolveLayerLayout()` 会：

- 至少补成 3 层
- 最多限制到 10 层
- 规范层类型为：
  - `CORE`
  - `URBAN`
  - `RING`
  - `BUFFER`
- 规范 density
- 自动补默认名称、默认生态、默认权重
- 保证第一层一定是 `CORE`
- 最后一层一定是 `BUFFER`

## 五、当前输出

创建成功后控制器会直接返回：

- `city_id`
- `actual_size`
- `blocks_total`
- `weight_sum`
- 每层的：
  - `type`
  - `weight`
  - `is_wall`
  - `chunks`
  - `blocks`
  - `ratio`

## 六、AI 决策和程序计算的边界

### 1. AI 决策的部分

按方案，C1 应该由 AI 决定：

- 城市候选点
- 最终中心点
- layers 结构
- 各 layer 的 weight
- 哪个 ring 是 wall 层
- `bias`
- `ecology`
- `density`

### 2. 程序计算的部分

- 校验中心点是否明显在水上
- 校验配置字段是否合法
- 标准化 layer 配置
- 把配置交给 `CityManager.createCity(...)`

## 七、当前和方案的差距

- 当前没有独立的 C1 候选扫描与选点流程。
- 现在更像“直接提交 CityConfig”，不是完整的候选决策阶段。
