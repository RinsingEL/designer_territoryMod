## T3 在当前项目里的真实分层

T3 当前是一条标准程序阶段：

1. `T3Stage.execute(...)` 负责触发
2. `TerritoryManager.runExpansion()` 负责真正扩张
3. `TerritoryResultRepository.exportAllT3(...)` 负责导出
4. `analyzeTerritories(...)` 负责生成摘要统计

AI 在 T3 运行时不再直接参与，只能通过 T1/T2 预先设置的参数影响结果。

## 三、程序具体怎么扩张

### 1. 建立统一扩张网格

`buildExpansionGrid(...)` 会：

- 以 `ScanResultHolder.lastScanData` 为 coarse map
- 如果 region detail cache 可用，则尽量下采样到更细步长
- 当前细扫优先常量是 `HIGH_RES_EXPANSION_STEP = 16`
- 先铺 coarse fallback，再用 detail cache 覆盖

结果会形成：

- `expansionScanMap`
- `expansionStepBlocks`
- `expansionMinX/minZ`

### 2. 修正每个领地的首都种子

每个 `TerritoryConfig` 的首都会先映射到扩张网格坐标。

如果原始 capital 不在合法大陆/区域内，`snapSeedToRegion(...)` 会寻找：

- 最近的合法 land cell
- 且 cluster id 必须等于该领地的 `regionId`

这样可以避免首都点落错大陆。

### 3. 使用多源 Dijkstra 扩张

`computeGlobal(...)` 的本体是多源 Dijkstra：

- 每个领地种子都进入优先队列
- 队列元素包含：
  - 当前累计 cost
  - 网格坐标
  - faction 索引

扩张限制：

- 必须有 `ScanPixel`
- 必须是 land
- 必须满足 `isCellAllowedForRegion(...)`
- 不能超出该 faction 的 `maxPower`

### 4. moveCost 如何计算

程序当前使用的扩张成本主要由坡度和水域构成。

公式大意：

- `moveCost = 1.0 + slope * mountainCost`
- `waterFactor = 1.0 + (1.0 - landRatio) * (waterCost - 1.0)`
- `moveCost *= waterFactor`

另外：

- 四邻方向成本系数是 `1.0`
- 斜向成本系数是 `1.414`

所以：

- `mountainCost` 越大，越不愿翻山
- `waterCost` 越大，越不愿过水或贴水扩张

### 5. claimArea 如何映射到 chunk

内部扩张不一定以 chunk 精度计算，但输出要落到 chunk。

`claimArea(...)` 会把扩张格反投影到真实世界坐标，再计算覆盖到哪些 chunk，然后写入：

- `claimedChunks`
- `wildChunks`

### 6. fillEnclaves 如何处理空洞

`fillEnclaves(...)` 会找出 `globalOwnershipMap` 里的空洞：

- 如果空洞不接触地图边界
- 且四周只被同一 sovereign 包围
- 就把它记为该领地的 `wildChunks`

### 7. analyzeTerritories 如何做统计

`analyzeTerritories(...)` 会为每个领地累计：

- `area_pixels`
- bbox
- `rawBiomeCounts`
- `rawContinentCounts`
- `neighborIds`

再转成最终摘要：

- `biome_composition`
- `continent_distribution`

邻国判断是按上下左右四邻格检查实现的。

## 四、阶段入口层逻辑

`T3Stage.execute(...)` 做的事情很直接：

1. `TerritoryManager.runExpansion()`
2. 读取 `getAllResults()`
3. `exportAllT3(...)`
4. 汇总 `claimedChunks + wildChunks`
5. 若总量为 0，则报错

这个报错会明确提示常见问题：

- W3/W4 扫描缓存缺失
- cluster map 缺失
- `territory.region_id` 和当前大陆 id 不匹配

## 五、输入

- T2 已确定的 territory config
- W3/W4 扫描缓存

## 六、输出

- T3 summary/stat 制品
- 运行时内存中的 territory results
- `globalOwnershipMap`

## 七、AI 决策和程序计算的边界

### 1. AI 决策的部分

AI 只在 T3 之前影响：

- `regionId`
- `capitalX/capitalZ`
- `maxPower`
- `mountainCost`
- `waterCost`

### 2. 程序计算的部分

- 建立扩张网格
- 修正种子点
- 运行 Dijkstra
- 判定 chunk 归属
- 计算 enclave
- 统计面积、邻国、群系、大陆分布

### 3. 当前代码和方案的差异

- `solution.md` 里提到未来可加入 biome preference。
- 当前 `TerritoryConfig` 还没有正式把 biome 偏好纳入扩张成本。
- 当前 moveCost 主要还是山地/水域惩罚。
