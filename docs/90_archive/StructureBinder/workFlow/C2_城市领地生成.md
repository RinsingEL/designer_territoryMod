# C2 城市领地生成

## 当前状态

- 状态：已实现，但语义拆成两半
- 需要特别注意：
  - 方案语义上的“城市占地与分层扩张”主要在 `CityManager.createCity(...)`
  - 名为 `handleCityC2Generate(...)` 的接口实际更像“占地之后的地形分析与局部细扫导出”

## C2 在当前项目里的真实分层

当前代码里的 C2 相关逻辑分成两段：

1. 城市占地与 layer 分配
2. 对已创建城市做基础地形分析和城市局部卫星扫描

## 一、城市占地与 layer 分配

### 入口

| 类 | 方法 | 说明 |
| --- | --- | --- |
| `server.mcp.CityController` | `handleCreateCity(HttpExchange)` | 接收 CityConfig |
| `world.city.CityManager` | `createCity(CityConfig)` | 真正生成城市占地 |

### 核心类与关键方法

| 类 | 方法 | 作用 |
| --- | --- | --- |
| `world.city.CityManager` | `createCity(...)` | 创建/覆盖城市实例 |
| `world.city.CityManager` | `expandCity(...)` | 核心扩张算法 |
| `world.city.CityManager` | `calculateCellCost(...)` | 计算向外扩张的单步成本 |
| `world.city.CityManager` | `assignLayersByWeight(...)` | 按权重分配 layer |
| `world.city.CityManager` | `computeBorderChunks(...)` | 计算边界 chunk |
| `world.city.CityManager` | `validateCityWithinSovereignty(...)` | 防止越界到别国 |
| `util.VoronoiComputer` | `computeDistricts(city)` | 生成 district |

### 程序具体怎么扩张

#### 1. 以中心 chunk 为起点

- 起点是 `centerX/centerZ` 所在 chunk
- 如果起点不在 territory sovereignty 内，直接报错

#### 2. 用优先队列向外扩张

`expandCity(...)` 使用按 cost 排序的优先队列扩张城市。

扩张时会跳过：

- 已被其他城市占用的 chunk
- 不属于该 territory sovereignty 的 chunk

#### 3. 单步成本怎么算

`calculateCellCost(...)` 当前主要考虑：

- `ScanPixel`
- `bias`
- 是否是水域

大致逻辑：

- 基础 cost = `1.0`
- 若扩张方向和 `bias` 相反，会加惩罚
- 若采样点在水里：
  - `COASTAL` 城市惩罚较小
  - 其他城市惩罚很大

#### 4. 城市规模怎么来

如果 `targetChunkCount <= 0`，会调用 `estimateTargetChunks(...)`：

- 依据 territory 的 `area_pixels`
- 近似公式：`area_pixels * 0.08`
- clamp 到 `80 ~ 520`

#### 5. layer 怎么分配

`assignLayersByWeight(...)`：

- 先把所有 claimed chunk 按扩张 cost 从低到高排序
- 再按 `weight` 计算各层 quota
- 成本最低的块优先给内层

所以当前 layer 分配是“成本中心化”的，不是纯几何环带。

#### 6. district 怎么生成

`createCity(...)` 末尾会调用 `VoronoiComputer.computeDistricts(city)`，因此 district 已经在占地结束后生成。

## 二、地形分析与局部细扫

### 入口

| 类 | 方法 | 说明 |
| --- | --- | --- |
| `server.mcp.CityController` | `handleCityC2Generate(HttpExchange)` | 输出 buildable 基础数据和城市卫星预览 |

### 核心类与关键方法

| 类 | 方法 | 作用 |
| --- | --- | --- |
| `world.NationGenManager.Stage1Manager` | `computeAndSave(...)` | 生成 Stage1 结果 |
| `world.city.stage.c1.CityStage1Processor` | `compute(...)` | 计算高度图、禁建点、可建组 |
| `server.mcp.CityController` | `computeCityScanBounds(...)` | 计算城市局部扫描范围 |
| `server.mcp.CityController` | `resolveC2ScanData(...)` | 依次走内存缓存/磁盘缓存/实时扫描 |
| `world.city.stage.c2.CityC2ScanBinaryIO` | `save(...)` / `load(...)` | 存取局部扫描 |
| `world.city.stage.c2.CityC2SatellitePreviewExporter` | `export(...)` | 输出卫星预览 |

### 程序具体怎么做

- 先计算并保存 Stage1：
  - 高度图
  - 坡度图
  - 粗糙度图
  - 禁建点
  - 可建组
- 再算扫描边界 `CityScanBounds`
- 通过 `resolveC2ScanData(...)` 优先使用缓存
- 实在没有才调用 `SatelliteScanner.scanRegionAsync(...)`
- 最后导出 `satellite_preview`

## 三、输入

- `city_id`
- 可选 `scan_step`
- 可选 `scan_padding_blocks`

## 四、输出

- Stage1 基础分析结果
- 城市局部扫描数据文件
- `satellite_preview`
- 控制器返回：
  - `district_count`
  - `buildable_group_count`
  - `scan_data_file`
  - `scan_cache_hit`
  - `scan_cache_source`

## 五、AI 决策和程序计算的边界

### 1. AI 决策的部分

- 来自 C1：中心点、layers、bias、ecology、density
- C2 运行时 AI 不直接参与扩张

### 2. 程序计算的部分

- 城市占地扩张
- 成本计算
- layer 配额分配
- district 生成
- 地形分析
- 局部细扫与预览导出

## 六、当前语义和方案语义的差异

- 按方案，C2 应该就是“城市领地生成”。
- 当前代码里，这件事已经在 `createCity(...)` 发生。
- 而 `city_c2_generate` 这个接口更像“C2 后置分析与预览导出”。
