# C7 工头计划实现

## 入口方法

| 位置 | 作用 |
| --- | --- |
| `CityController.handleCityC7Generate(...)` | 解析请求、选择 `generate` 或 `fromDecisionRequest`、按 group 落盘 |
| `CityController.handleCityC7Data(...)` | 读取城市级或 group 级 `C7` 文件 |

## 主要实现点

| 位置 | 说明 |
| --- | --- |
| `CityC7Stages.generate(...)` | 程序 fallback 生成工头计划 |
| `CityC7Stages.fromDecisionRequest(...)` | 接收外部 `arrangements` 决策并转换为 `foreman_plan` |
| `CityC7Stages.buildForemanPlan(...)` | 组装 `ForemanPlan / StartNode / PhasePlan` |
| `CityC7Stages.chooseCandidates(...)` | 结合 catalog 和过滤规则选择候选模板 |
| `CityC7Stages.finalizeSelectionResult(...)` | 整理兼容字段与主视图 |

## 当前落盘路径

| 场景 | 路径 |
| --- | --- |
| 城市级 | `cityDir/C7_TemplateSelection.json` |
| group 级 | `groupDir/c7_selection.json` |

## 当前实现细节

- `CityController.handleCityC7Generate(...)` 会优先读取 `strict_tag_source`，再决定是否启用严格过滤。
- 当请求体带 `arrangements` 时，入口不再走纯程序 fallback，而是调用 `fromDecisionRequest(...)`。
- group 级生成后会追加 `CityC7Validation.generate(...)` 与 `GroupStepStateUtil.update(...)`，把校验失败和阻塞态一起回写。

## 继续追代码建议

1. 先看 `handleCityC7Generate(...)` 的回包字段。
2. 再看 `buildForemanPlan(...)` 如何把 `start_node / phase_list / template_pool_refs` 固定下来。
3. 如需理解候选模板为什么被选中，再回看 `chooseCandidates(...)` 与 `StructureTemplateQueryService`。
