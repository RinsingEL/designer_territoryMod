## T1 当前流程

1. `T1Stage.dependsOn()` 声明依赖 W4。
2. `TerritoryStageOrchestrator.runT1(...)` 读取 blueprint 列表。
3. 若 `ScanResultHolder.regionCacheMap` 为空，直接报错阻塞。
4. 按 continent 分组处理每个 `PlanEntry`。
5. `prepareT1(...)` 生成候选 bundle、过滤冲突、写出 `T1_Candidates.json` 与 `T1_Status.json`。
6. `selectT1Cluster(...)` 在 AI 或用户选择后固化簇，并删除旧 T2 产物。

T1 只锁定候选簇，不创建 territory config。

## T2 当前流程

1. `T2Stage.dependsOn()` 声明依赖 T1。
2. `prepareT2(...)` 读取 T1 状态与候选 bundle。
3. 若 T1 未完成或候选不可用，T2 状态进入 blocked。
4. 生成方向候选并写出 `T2_DirectionCandidates.json`。
5. `selectT2Direction(...)` 校验所选方向可用。
6. `applyDirectionSelection(...)` 根据所选点调用 `TerritoryManager.createTerritory(...)`，写出 T2 状态与首都摘要。

T2 是 T3 的直接门禁：T3 readiness 会检查 T1 已选簇、T2 已选点、territory config 已写入。

## T4 当前流程

1. `T4Stage.dependsOn()` 不再直接声明依赖 T3，显式执行 T4 时不会自动重跑 T3。
2. `T4Stage.runLegacyTerrainScan(...)` 先调用 `TerritoryManager.restoreT3ResultsFromDisk(ctx.server)`，把当前实例 id 的 canonical T3 结果恢复到运行态。
3. 读取 `TerritoryManager.getAllResults()`，对每个国度合并 `claimedChunks` 与 `wildChunks`。
4. 按 `RuntimeOptions` 采样地形高度、温度、biome、坡度、边境距离与首都距离。
5. 通过 `TerritoryResultRepository.writeT4(...)` 写出每个国度的 T4 summary/dat。
6. 写出 batch report，并在所有国度失败时抛错。

T4 有两种采样路径：

- `loaded_only=false`：从 generator/noise 读取，不强制加载 chunk。
- `loaded_only=true`：只读已加载 chunk，自动触发时可降低卡顿风险。

## 查询与调试入口

- `/territory_status`：读取当前 territory config，并在响应前尝试把当前实例 id 的 T3 结果从磁盘恢复到运行态。
- `/territory/summary`：优先读 artifact，缺失时尝试读内存里的 T3 summary。
- `/territory/t4_window`：从 T4 dat 解码局部窗口，给 C1 或人工检查使用。未显式传 center 时会先取 capital；如果 capital 不在 T4 覆盖区，会自动 reanchor 到最近 T4 采样点并回传 warning。
- `/territory/t3_import`：把历史基础 id 的 T3 结果显式导入到当前实例 id，同时把导入结果同步进运行态内存。

## 常见失败点

- T1 报 W4 数据缺失：目标大陆未完成 W4 region scan。
- T2 报 T1 candidates 未生成：需要先跑 T1 或 `/t1_candidates_for_continent`。
- T2 所选方向不可用：方向候选里 `available=false`，通常是 seed cell 冲突。
- T3 readiness 阻塞：T1/T2 未完成、territory config 缺失、首都点或 seed cell 重复。
- T4 报没有 territory results：当前实例没有有效 canonical T3，可先完成 T3 或用 `/territory/t3_import` 导入历史 T3。
- T3 零面积回归：新的零面积结果不会覆盖已有非零 canonical T3，而会写到 `T3_failed` 诊断目录。
