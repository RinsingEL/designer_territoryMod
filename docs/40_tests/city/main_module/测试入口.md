# 主模块建造测试入口

## 当前测试目标

主模块系统当前重点验证三件事：

1. `C7 -> C8` 的计划与会话主链是否稳定。
2. `C8` 节点提交、失败、重试是否可持续推进。
3. `city_jigsaw_solve` 是否能输出可读证据，并在需要时正确衔接执行层。
4. 地表清理与落地后 jigsaw 收口是否符合当前执行口径。

## 功能级测试清单

| 编号 | 功能 | 测试项 | 预期结果 |
| --- | --- | --- | --- |
| M1 | C7 工头计划 | 调用 `city_c7_generate` 生成 group 计划 | 成功生成 `c7_selection.json`，含 `foreman_plan / phase_list / start_node` |
| M2 | C7 工头计划 | group 模式校验失败 | 返回 `422` 或 `409`，并回包校验信息 |
| M3 | C8 会话生成 | 调用 `city_c8_generate` | 成功生成 `c8_foundation.json`，返回 `session_state / active_node / queue_summary` |
| M4 | C8 数据读取 | 调用 `city_c8_data` | 能读到最新 group 级会话 |
| M5 | C8 节点提交 | 对 `active_node` 发起 `city_c8_submit` | 成功写入 `validated_nodes`，并回包 `submit_result / queue_summary` |
| M5.1 | C8 节点立即落地 | 对 `active_node` 发起 `city_c8_submit` 且 `apply_now=true` | 当前节点先完成 `C8` 校验，再立即进入执行层；回包 `placement_execution`，并能在世界里直接看到 start / 普通节点落地结果 |
| M6 | C8 失败重试 | 对失败节点调用 `city_c8_retry` | 节点回到可继续推进状态，`active_node` 同步更新 |
| M6.5 | Group 地表清理 | 调用 `city_group_surface_clear` | 默认不传 `min_y` 时应从 `foundation.base_y - 1` 开始清；显式传 `min_y / max_y` 时严格按请求值执行 |
| M7 | Jigsaw 求解 | 调用 `city_jigsaw_solve` | 生成 `trace.json` 与步骤预览，返回结构化 `solve_result`，并把成功 child 写回 `validated_nodes / active_node` |
| M8 | Jigsaw 立即落地 | `city_jigsaw_solve` 开启 `apply_now` | 返回 `apply_result`，并能在执行层产物中看到结果同步；`post_cleanup.reason_counts` 能区分 `jigsaw_final_state / jigsaw_air_fallback` |
| M8.1 | C8 去重与续链 | 连续执行 `start -> child -> child` | `validated_nodes / placements` 不再重复累计同名 `start`，且成功 child 后会继续生成下一层 `active_node / node_queue` |
| M8.2 | runtime pool 候选收束 | 验证 solved child / grandchild 后的 `template_pool_id / candidate_template_ids` | `minecraft:village/plains/houses` 不再回成整包 `village_misc`；若 runtime 与 catalog 冲突，`trace.json` 与 `node_debug` 中能看到 mismatch |
| M8.3 | Jigsaw 调试图可读性 | 打开 `jigsaw_solver_debug/<runId>/0*.png` 与 `*.legend.json` | 标签应为外置错位、半透明底、带引线；legend 应带 `label_layout = external_staggered` |

## 当前证据入口

| 类型 | 位置 |
| --- | --- |
| `C7` 产物 | `group/<groupId>/c7_selection.json` |
| `C8` 产物 | `group/<groupId>/c8_foundation.json` |
| `GroupSurfaceClear` 产物 | `group/<groupId>/group_surface_clear/<runId>.json` |
| `Jigsaw` 调试产物 | `group/<groupId>/jigsaw_solver_debug/<runId>/` |
| 下游执行结果 | `group/<groupId>/c9_placement.json` 与 `C9_BuildQueue` |

## 建议联调顺序

1. 先跑 `city_c7_generate`
2. 再跑 `city_c8_generate`
3. 对 `active_node` 反复执行 `city_c8_submit / city_c8_retry`
4. 如需校验清地口径，先调用 `city_group_surface_clear`
5. 如需把当前节点直接放到世界里，对同一节点使用 `city_c8_submit` 并带 `apply_now=true`
6. 需要 child 结构时再进入 `city_jigsaw_solve`
7. 对同一 parent 成功执行一次 `city_jigsaw_solve` 后，应确认：
   - `start` 这类 runtime start 已自动归一化到正确高度
   - 旧 placeholder child 已被真实 `vjigsaw_*` child 替换
   - `active_node` 已切到真实 child 的下一层节点
   - 下一层 `template_pool_id / candidate_template_ids` 已按 runtime pool 收束，而不是回成整包逻辑池
   - 若存在 runtime/catalog pool 冲突，`solve_result.placement.outgoing_connectors` 与 `node_debug` 中可见 mismatch 诊断
8. 涉及立即落地时，再回看 `C9` 执行层系统文档
