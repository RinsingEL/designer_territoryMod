## 系统级文档组织规则

从当前阶段开始，系统文档 `MUST` 逐步按“一个系统一个文件夹”维护。

### 目录组织要求

同一系统应尽量在以下层级使用同名目录：

- `docs/10_product/<领域>/<系统>/`
- `docs/20_contracts/<领域>/<系统>/`
- `docs/30_code_guide/<领域>/<系统>/`
- `docs/40_tests/<领域>/<系统>/`

agent `MUST` 遵守：

1. 一个系统只维护一套当前有效目录，不把多个系统揉在同一份主文档中。
2. 新建系统文档或重构旧文档时，`MUST` 优先采用目录化结构，不再新增平铺式系统主文档。
3. 旧文件如 `01_scope.md`、`01_acceptance.md`、平铺式 `main_module.md` 可暂时保留，但当本轮任务触达该系统当前真值时，`MUST` 同步补齐或迁移到新结构。

### 最小文档骨架

任一系统至少应逐步具备以下文档：

- `docs/10_product/<领域>/<系统>/系统概述.md`
- `docs/10_product/<领域>/<系统>/功能设计/<功能名>.md`
- `docs/20_contracts/<领域>/<系统>/配置表/<对象或功能名>.md` 或等价契约文档
- `docs/30_code_guide/<领域>/<系统>/代码导览.md`
- `docs/40_tests/<领域>/<系统>/测试入口.md`
- `docs/40_tests/<领域>/<系统>/影响面.md`

若系统还在起步阶段，可先补以下最小集合：

- `系统概述.md`
- 本次目标功能的 `功能设计/<功能名>.md`
- `影响面.md`

但只要已经出现结构化数据、代码入口或验收路径，就 `MUST` 继续补齐对应层级文档。

### 写作要求

agent 在本仓库写系统文档时，`MUST` 优先遵守以下原则：

1. `系统概述.md` `MUST` 说明系统目标、责任边界、上下游依赖、核心子功能和文档导航。
2. `功能设计/<功能名>.md` `MUST` 优先说明功能目标、入口条件、核心流程、异常边界和关联文档。
3. 文档中只要存在流程，`SHOULD` 优先使用 Mermaid 流程图。
4. 文档中只要存在数据结构、状态枚举、参数说明、配置驱动逻辑，`SHOULD` 优先使用 Markdown 表格或配置表形式。
5. 文档表达 `MUST` 简洁，不把实现细节写成冗长散文。

## 双仓库协作规则

当前双仓库协作约定如下：

- 实现仓库负责：
  - Java / MCP / 运行时实现
  - 调试、联调、`dev_docs` 过程记录
- 文档仓库负责：
  - 功能方案真值
  - 契约真值
  - 代码导览真值
  - 测试与验收入口真值
  - 系统影响面真值

当实现仓库发生以下变化时，agent `MUST` 判断是否同步更新本仓库：

- 阶段职责变化
- 接口或数据结构变化
- 测试入口或验收方式变化
- 代码入口、主链路径或类职责变化
- 功能边界、系统拆分方式、功能流程或影响面变化

## 影响面文档维护规则

每次系统开发时，agent `MUST` 同步维护：

- `docs/40_tests/<领域>/<系统>/影响面.md`

维护方式 `MUST` 遵守：

1. `影响面.md` 是持续维护的一份总文档，不按时间戳不断新建副本。
2. 文档内部按功能小标题维护，例如 `## 图鉴功能`、`## 道路生成`。
3. 同一功能每次改动都回到同一个小标题下更新，不新增并行标题来替代历史脉络。
4. 每个功能小标题至少写清：
   - 影响的模块、接口、配置、阶段
   - 影响的测试入口或验收口径
   - 影响的上下游系统
   - 当前风险、限制或未决问题

## 自测问题与文档联动规则

当 AI 在联调、自测、验收中发现问题时，agent `MUST`：

1. 明确指出问题现象、卡点位置和影响范围。
2. 指向建议优先复查的系统文档或功能文档。
3. 判断下一步是：
   - 继续由 AI 补日志、自查、自修。
   - 还是因为设计偏离、文档冲突、边界不清，需要人工介入。
4. 如果问题暴露出当前文档已失真，`MUST` 优先修正文档，再继续推进实现或测试。

## Git 提交规则

本仓库中的 agent 在执行 git 提交时，必须遵守以下规则：

1. `MUST` 使用全中文提交信息。
2. `MUST` 在提交标题中明确概括本次文档变更主题，不得使用“更新”“修改”“fix”这类模糊标题。
3. `MUST` 在提交正文中说明：
   - 更新了哪些功能文档
   - 更新了哪些契约或导览
   - 是否涉及影响面文档维护
   - 是否涉及历史归档调整
   - 与实现仓库的对应关系
4. `NEVER` 提交未经确认的实现仓库文件镜像或无关资料。

## 写作规则

agent 在本仓库写文档时，`MUST` 优先遵守以下原则：

1. 当前真值优先写“现在怎么做”，不要把历史讨论原样搬成主文档。
2. 产品方案、契约、代码导览、测试入口、影响面必须分层，不混写成一份大杂烩。
3. 从历史资料抽取内容时，要重写成当前口径，而不是简单复制旧文档。
4. 每份文档尽量能让读者快速回答三个问题：
   - 这块现在的目标是什么
   - 它和实现仓库的哪个部分对应
   - 如果要继续开发，下一步应该看哪里
